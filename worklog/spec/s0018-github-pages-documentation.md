## Behavior

- GitHub Pages hosts generated API documentation for `stelaro` and all `stelaro-*` packages in the monorepo.
- Documentation is generated by TypeDoc from the public entry points of each package.
- A `build-docs` script at the workspace root generates documentation into `docs/`.
- `docs/` contains a `.nojekyll` file so GitHub Pages serves the output as-is.

### CI/CD

- Documentation is automatically built and deployed to GitHub Pages on every push to `main`, after build-and-test succeeds.
- Pushes that only touch `docs/**` do not trigger CI. `UNIMPLEMENTED`

### Scope

- Documentation covers all packages under `packages/*` — currently `stelaro`, `stelaro-discord`, `stelaro-fastify`, `stelaro-pino`.
- When a new `stelaro-*` package is added, its public entry point must be included in the TypeDoc configuration.

## Constraints

- No custom domain — deployed to the default `<user>.github.io/stelaro` path.
- Generated `docs/` is not committed to the repository; deployment is artifact-based (upload → deploy).
- TypeDoc entry points must match the actual package structure, not a hardcoded glob.

## Anticipated Changes

- Authored documentation (guides, getting started) alongside generated API reference.
- Versioned documentation for stable releases.
- Search integration.

## Dangers

- Adding a package without updating TypeDoc entry points silently omits it from documentation.
- TypeDoc version drift across monorepo packages could produce inconsistent output.
