## Behavior

- UNIMPLEMENTED Peranto supports Hot Module Replacement during development.
- UNIMPLEMENTED Hot Module Replacement updates application behavior without requiring a full application restart.
- UNIMPLEMENTED Hot Module Replacement uses stable component ids to identify replacement targets.
- UNIMPLEMENTED Hot Module Replacement may preserve in-memory component state when explicitly supported.

## Constraints

- Hot Module Replacement behavior is development behavior, not required production behavior.
- Hot Module Replacement must not weaken component id stability.
- Hot Module Replacement must not silently ignore incompatible replacements.
- State preservation during Hot Module Replacement must be explicit.

## Anticipated Changes

- Component replacement lifecycle behavior may be specified later.
- Gateway replacement behavior may be specified later.
- State preservation and migration behavior may be specified later.

## Dangers

- Preserving state implicitly can hide incompatible state changes.
- Replacing gateway behavior without clear lifecycle rules can leave stale external bindings active.
- Treating Hot Module Replacement as production lifecycle behavior can make application semantics harder to reason about.
